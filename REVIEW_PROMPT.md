# Промт для запуска реализации плана в новом диалоге

Скопируй текст ниже целиком в новый диалог (Claude Opus / Codex / другой
агент). Промт самодостаточный: содержит контекст, правила работы и ссылку на
главный план.

## START PROMPT

Ты работаешь над репозиторием ZI_RAG - RAG-сайдкар для OpenWebUI.

**Корень проекта:** `/media/onel/videoanalytics/ZI_RAG`

**Структура:**
- `openwebui_zi_rag/` - Python-пакет сайдкара (FastAPI + FAISS + SQLite).
  - `server.py` - эндпоинты и orchestration, сейчас больше 2000 строк.
  - `config.py` - `SidecarConfig` (dataclass, env+json merge).
  - `ollama_client.py` - клиент Ollama и OpenAI-compatible embeddings.
  - `indexing/` - `registry.py` (SQLite), `extraction.py` (parsers + OCR),
    `chunking.py`, `vector_store.py` (FAISS), `service.py` (RagService).
  - `web/` - UI админки.
- `openwebui_functions/zi_rag_filter.py` - Function-фильтр для OpenWebUI
  (single-file, копируется в OpenWebUI Admin Panel).
- `openwebui_functions/zi_rag_filter.openwebui.json` - JSON-экспорт того же
  фильтра для OpenWebUI.
- `tests/test_openwebui_zi_rag.py` - pytest-тесты.
- `REVIEW_PLAN.md` - **главный план работ. Читай его первым.**

**Главное правило:** все задачи берутся из `REVIEW_PLAN.md`. Каждый пункт имеет
id (1.1, 2.3, 4.6 и т.д.) и статус (`[ ]`, `[~]`, `[x]`, `[?]`). Пункты `[x]`
не реализуй заново: например, 1.2 закрыт как not-a-bug.

### Первый шаг

1. Открой `REVIEW_PLAN.md`.
2. Выведи список открытых пунктов со статусами.
3. Предложи, с каких пунктов начать, обычно с high-priority раздела 1.
4. Не правь код до моего подтверждения выбранных пунктов.

### Что делать в каждой итерации

1. Перед правкой пункта поставь в `REVIEW_PLAN.md` статус `[~]`.
2. Реализуй минимальный scoped fix по выбранному пункту.
3. Напиши или обнови тесты из раздела 7, соответствующие закрытому пункту.
4. Запусти локально:
   `python3 -m pytest tests/test_openwebui_zi_rag.py --no-header -x`
   Если в окружении есть только другая команда Python, явно сообщи это.
5. После успешной проверки поставь `[x]` в плане и добавь строку в
   `## История изменений` с датой и кратким описанием.
6. Если пункт требует проверки на стенде, поставь `[?]` и допиши конкретную
   инструкцию ручной проверки.
7. Если по ходу работы найдена новая проблема, добавь новый пункт в
   соответствующий раздел плана. Не реализуй новый пункт без отдельного
   подтверждения, кроме случаев, когда он блокирует выбранный fix.

### Git и окружение

- В текущей копии `/media/onel/videoanalytics/ZI_RAG` может не быть `.git`.
  Сначала проверь `git status`. Если это не git-репозиторий, не пытайся делать
  коммиты и просто обновляй файлы.
- Если git есть, не откатывай чужие изменения. Коммить только если я явно
  прошу коммит или если мы отдельно согласовали commit workflow.
- Не запускать долгоживущие процессы (`uvicorn`, `npm run dev` и т.п.) в одном
  блокирующем shell-вызове.
- Не добавлять в коммит или bundle содержимое `openwebui_zi_rag_storage/`,
  FAISS-индексы, SQLite БД, OCR-модели, `.pytest_cache/`, `__pycache__/`.
- `openwebui_zi_rag_bundle.zip` не править вручную. Если нужна пересборка
  bundle, сначала завести/использовать отдельный build-скрипт.

### Жёсткие ограничения

- **`openwebui_functions/zi_rag_filter.py` должен оставаться single-file.**
  Это требование OpenWebUI Functions.
- Пока нет `tools/build_filter.py`, любая правка
  `openwebui_functions/zi_rag_filter.py` должна быть вручную синхронизирована с
  `openwebui_functions/zi_rag_filter.openwebui.json`.
- При правках фильтра добавь/сохрани regression test синхронизации `.py` и
  `.openwebui.json` из п. 7.14 плана.
- Общий код с сайдкаром можно выносить в `openwebui_zi_rag/text_utils.py`, но
  фильтр всё равно должен содержать копию нужного кода.
- Не ломать имена полей `config.json`, схему SQLite и API эндпоинтов без явного
  согласования. Установка модуля идёт с нуля, но совместимость имен важна для
  документации и UI.
- При правке `server.py` не делать мегарефактор одним шагом. Декомпозиция
  (п. 2.1) идёт отдельным этапом после high-priority багов 1.1-1.10.

### Зафиксированные решения

- **Generation model (8.1):** хардкод `"qwen3.6:latest"` убрать полностью.
  Дефолт пустая строка. Пользователь выбирает модель в UI из `/ollama/models`.
  Если модель не выбрана или отсутствует в `/api/tags`, вернуть 409 с понятным
  сообщением и списком доступных моделей. Обновить также `OPENWEBUI_ZI_RAG.md`;
  `qwen3.6:latest` в тестах считать тестовым именем, не production fallback.
  Для Compliance допустим явный fallback на `deep_generation_model`, если
  `compliance_generation_model` пустой.
  См. п. 1.5.
- **Strict api_key для localhost (8.2):** добавить опциональный
  `require_api_key_localhost: bool = False`. По умолчанию поведение не меняется.
  Включается чекбоксом в UI и env `ZI_RAG_REQUIRE_API_KEY_LOCALHOST=1`.
  Strict без `api_key` должен отклоняться как некорректный конфиг. См. п. 3.1.
- **FAISS-индекс (8.3):** `index_type=auto`, Flat до 50k чанков, HNSW выше.
  Параметры HNSW в конфиг. Сохранить текущую нормализацию и scoring. См. п. 4.1.
- **Pydantic v2 BaseSettings (8.4):** допустим, но нужен `pydantic-settings` в
  requirements и отдельный этап. Миграция старого `config.json` не нужна, имена
  полей сохранить. См. п. 2.3.

### Стиль кода

- Python 3.10+, типизация через `dict[str, Any]`, `list[...]`, `|`.
- `from __future__ import annotations` уже стоит - сохранять.
- Никаких новых зависимостей без обоснования. Исключение уже согласовано для
  п. 2.3: `pydantic-settings`, но добавлять его только при реализации 2.3.
- Сообщения логов и UI - на русском, пока не реализован п. 6.4 (i18n).
- Атомарная запись файлов - через tmp + `os.replace`, кроме сценариев, где
  важно не перетереть чужой файл. Для `save_upload` из п. 1.7 нужен
  `O_CREAT | O_EXCL` с уникальным suffix.
- Все SQL-параметры - placeholder'ы, без f-string интерполяции значений.
- Для structured data использовать `json`, sqlite placeholders и существующие
  helper APIs, а не ad-hoc string parsing.

### Тесты

- Основной запуск:
  `cd /media/onel/videoanalytics/ZI_RAG && python3 -m pytest tests/test_openwebui_zi_rag.py --no-header -x`
- Покрывать каждый закрытый баг минимум одним тестом из раздела 7.
- Не использовать сетевые/GPU/Ollama-вызовы в тестах: подменять
  `OllamaClient` / `OpenAIEmbeddingClient` фейками. В test-файле уже есть
  `FakeEmbeddingClient`, `CountingEmbeddingClient`, `CapturingEmbeddingClient`.
- Для concurrency-тестов держать таймауты короткими и детерминированными.

## Как пользоваться

4. Проверить, что агент действительно прочитал план: попросить вывести список
